## Overview

The e2e-encryption component provides end-to-end encryption between the coordinator and provider's hardened process using NaCl Box (X25519 + XSalsa20-Poly1305). It ensures that inference request payloads are encrypted with the provider's X25519 public key (obtained from their Secure Enclave attestation) before being sent over WebSocket, so only the provider's hardened binary can decrypt them — not even if the provider runs a MITM proxy on their own network. This component is the critical privacy layer that prevents the machine owner from observing user prompts or model responses.

## Architecture

- **`coordinator/internal/e2e/e2e.go`**: Core Go package implementing the coordinator-side encryption. Defines the `EncryptedPayload` struct (containing `EphemeralPublicKey` and `Ciphertext` as base64 strings), the `SessionKeys` struct (holding ephemeral `[32]byte` key pairs), and the `GenerateSessionKeys` function for creating per-request ephemeral key pairs. Uses `golang.org/x/crypto/nacl/box` for NaCl Box operations.

- **`decrypt-test` (Rust CLI)**: A minimal Rust binary (`src/main.rs`) that validates cross-language cryptographic compatibility. Takes three CLI arguments (ephemeral public key, ciphertext, provider private key), performs base64 decoding, validates input sizes (32-byte keys, minimum 24-byte ciphertext for nonce + data), constructs a `SalsaBox`, and attempts decryption. Built via `cargo build --release` during Go test execution.

## Dependencies

- **Provider Secure Enclave Attestation (EigenInferenceEnclave)**: The e2e component depends on the provider's X25519 public key, which is included in the attestation blob generated by the Secure Enclave. Without this public key, the coordinator cannot encrypt payloads for a specific provider.

- **`golang.org/x/crypto/nacl/box`**: External Go library providing the NaCl Box implementation (X25519 + XSalsa20-Poly1305) on the coordinator side.

- **Rust `crypto_box` crate**: Provider-side dependency for decryption. The `decrypt-test` component uses this to verify interoperability with the Go implementation.

## API Surface

- **`EncryptedPayload` struct**: JSON-serializable type with `EphemeralPublicKey` (base64-encoded sender ephemeral X25519 public key) and `Ciphertext` (base64-encoded 24-byte nonce || encrypted+authenticated data). This is the primary data structure transmitted over WebSocket.

- **`SessionKeys` struct**: Holds `[32]byte` `PublicKey` and `PrivateKey` for an ephemeral key pair created per inference request.

- **`GenerateSessionKeys()`**: Creates a new ephemeral X25519 key pair for forward secrecy on each request.

## Security Considerations

- **Forward Secrecy**: The coordinator generates a fresh ephemeral X25519 key pair per inference request, ensuring that compromise of a single session key does not expose past or future communications.

- **Authenticated Encryption**: NaCl Box (XSalsa20-Poly1305) provides both confidentiality and integrity/authentication, preventing tampering with ciphertext.

- **Provider Private Key Isolation**: The provider's private key never leaves the hardened process. It is held only in protected memory within the runtime-hardened binary where debuggers are denied at the kernel level (PT_DENY_ATTACH) and memory-reading APIs are blocked by Hardened Runtime.

- **MITM Resistance**: Even if the provider runs a MITM proxy on their own network, they cannot decrypt the payload without the private key held inside the hardened process.

- **Input Validation**: The Rust decrypt-test component enforces strict validation — ephemeral public key and provider private key must be exactly 32 bytes (X25519 standard), and ciphertext must be at least 24 bytes (nonce + encrypted data minimum).

- **No Key Persistence**: Private keys are only held in memory during execution; no keys are persisted to disk.

- **Constant-Time Operations**: The `crypto_box` library incorporates timing attack mitigations.

## Data Flow

1. **Key Exchange Setup**: The coordinator obtains the provider's X25519 public key from their Secure Enclave attestation blob during provider registration/attestation.

2. **Per-Request Encryption (Coordinator → Provider)**:
   - Coordinator generates ephemeral X25519 key pair via `GenerateSessionKeys()`
   - Derives shared secret: ephemeral private key + provider public key
   - Encrypts prompt using NaCl Box with the derived shared secret
   - Constructs `EncryptedPayload` containing the ephemeral public key (base64) and ciphertext (base64: 24-byte nonce || encrypted+authenticated data)
   - Sends `EncryptedPayload` over WebSocket to the provider

3. **Decryption (Provider Side)**:
   - Provider receives `EncryptedPayload` over WebSocket
   - Base64-decodes the ephemeral public key and ciphertext
   - Derives the same shared secret: provider private key + ephemeral public key
   - Decrypts ciphertext using NaCl Box, extracting the plaintext prompt
   - Executes inference on the decrypted prompt within protected memory

4. **Response Encryption (Provider → Coordinator)**:
   - Provider encrypts the inference response using: provider private key + ephemeral public key
   - Sends encrypted response back over WebSocket

5. **Response Decryption (Coordinator)**:
   - Coordinator decrypts the response using: ephemeral private key + provider public key
   - Returns plaintext response to the consumer
